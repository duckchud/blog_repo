<p style="font-size:1.15em; color:#4b5563;">LLM에게 지식을 직접 외우게 하지 말고, <strong>외부 온톨로지 그래프</strong>를 장기 기억·검증·계획 계층으로 붙이자는 논문을 정리했다.</p>

> 📄 원 논문: *Automatic Ontology Construction Using LLMs as an External Layer of Memory, Verification, and Planning for Hybrid Intelligent Systems* ([arXiv](https://arxiv.org/abs/2604.20795))

## 이 논문을 보는 두 축

구조적으로 보면 이 논문의 핵심은 두 축으로 요약된다.

<div style="margin:16px 0; padding:12px 16px; border-left:4px solid #3b82f6; border-radius:8px; background:#eff6ff; color:#1f2937; line-height:1.6;">
🧱 <strong>① Dual Storage</strong> — 텍스트 기억과 구조 기억을 분리한다. <em>(기억의 구조)</em><br>
🔁 <strong>② Feedback Loop</strong> — 생성된 답변과 새 정보를 다시 구조 기억으로 환류한다. <em>(기억을 갱신하는 작동 원리)</em>
</div>

이중 저장이 "기억의 구조"라면, Feedback Loop는 그 기억을 **계속 갱신하는 작동 원리**다. 논문은 자동 온톨로지 구축을 일회성 변환 작업이 아니라, 답변과 대화에서 새로운 사실을 추출·검증한 뒤 그래프를 지속적으로 업데이트하는 **폐쇄형 과정**으로 설명한다. 공식 초록 역시 `continuous graph updates`와 `generation–verification–correction pipeline`을 핵심 기여로 제시한다. ([arXiv](https://arxiv.org/abs/2604.20795))

# Part 1. Dual Storage — 기억의 구조

## 1. 핵심 요약

LLM이 지식을 직접 기억하고 판단하도록 두기보다, 외부의 온톨로지 그래프를 장기 기억·검증·계획 계층으로 사용해 시스템의 신뢰성과 설명 가능성을 높이자는 논문이다.

이 논문은 원문 전체를 온톨로지로 변환하지 않는다. 모든 텍스트를 그래프로 구조화하려 하면 정보 손실이 발생하고 구축 비용도 커지기 때문이다. 대신 다음과 같은 **이중 저장 구조**를 사용한다.

- 원문·문맥·로그 → **Vector Store**
- 엔티티·관계·규칙·상태 → **RDF/OWL Ontology Graph**

원문은 벡터 저장소에 보존하고, 그중 구조적으로 중요한 사실과 관계만 온톨로지 그래프로 변환한다. 질문이 들어오면 두 저장소의 검색 결과를 다시 결합한다.

## 2. 이중 저장 구조가 중요한 이유

일반적인 RAG는 문서를 청크로 분할하고, 각 청크를 임베딩으로 변환해 벡터 DB에 저장한다.

> 문서 → 청크 분할 → 임베딩 → Vector DB → 의미적으로 유사한 텍스트 검색

벡터 검색은 질문과 표현이 달라도 의미적으로 관련된 문서를 찾는 데 강하다. 하지만 다음과 같은 정보는 명시적으로 처리하기 어렵다.

- 엔티티 사이의 관계
- 규칙의 적용 조건
- 시간에 따른 상태 변화
- 다단계 관계 경로
- 기존 지식과 신규 사실의 모순
- 필수 속성과 제약조건

그래서 논문은 지식을 두 가지 형태로 동시에 보존한다.

| 저장소 | 저장하는 정보 | 주요 역할 |
| --- | --- | --- |
| **Vector Store** | 원문 청크, 설명, 로그, 대화, 임베딩 | 의미 기반 검색, 높은 회상률, 원문 보존 |
| **RDF/OWL Graph** | 엔티티, 관계, 타입, 규칙, 상태, 제약 | 정확한 관계 조회, 추론, 검증, 설명 가능성 |

논문은 벡터 저장소를 **recall과 textual residue**를 담당하는 계층으로, 온톨로지 그래프를 **semantics, constraints, explainability**를 담당하는 계층으로 설명한다.

- **RDF**: 지식을 *주어–관계–목적어* 형태의 그래프로 표현
- **SHACL**: RDF 그래프가 정의된 조건을 만족하는지 검증하는 표준

## 3. 같은 원문이 두 저장소에 저장되는 방식

예를 들어 다음 문장이 있다고 하자.

> Product A can be installed with Product B only when the clearance is at least 100 mm.

**Vector Store** — 원문과 문맥, 출처가 그대로 저장된다.

```json
{
  "text": "Product A can be installed with Product B only when the clearance is at least 100 mm.",
  "source": "installation_manual.pdf",
  "page": 14,
  "embedding": [ ... ]
}
```

**Ontology Graph** — 원문에서 추출한 구조화된 의미가 저장된다.

```
ProductA               compatibleWith            ProductB
Installation           requiresMinimumClearance  100mm
CompatibilityCondition appliesTo                 ProductA–ProductB
```

두 저장소는 같은 정보를 중복 보관하는 것이 아니라, **서로 다른 형태로 기억**한다.

- **Vector DB** → 원문과 근거를 보존하고 검색하는 기억
- **Ontology Graph** → 원문의 의미, 관계, 규칙을 운영하는 기억

따라서 서로 다른 종류의 질문을 처리한다.

- **Vector DB**: "설치와 관련된 문서나 유사한 내용을 찾아줘."
- **Ontology Graph**: "Product A와 Product B는 호환되는가?", "호환을 위해 필요한 조건은 무엇인가?"

## 4. 이중 저장의 세 가지 원칙

<div style="margin:16px 0; padding:12px 16px; border-left:4px solid #10b981; border-radius:8px; background:#ecfdf5; color:#065f46; line-height:1.6;">
✅ ① 원문은 버리지 않는다.<br>
✅ ② 구조화 가능한 핵심 의미는 그래프로 분리한다.<br>
✅ ③ 질문 시 두 저장소의 결과를 다시 결합한다.
</div>

즉, 그래프가 텍스트를 대체하는 것이 아니라 두 표현을 **상호 보완적**으로 사용한다.

## 5. 질의가 들어왔을 때의 동작

사용자가 이렇게 묻는다고 하자.

> "이 제품을 우리 집에 설치할 수 있고, 기존 제품과도 호환돼?"

시스템은 여러 기억 계층을 조회한다.

- **Vector Retrieval** — 설치 매뉴얼, 사용 설명서, 제품 FAQ, 과거 상담 기록
- **Graph Retrieval**
  - `ProductA hasWidth 680mm`
  - `ProductA compatibleWith ProductB`
  - `ProductA requiresClearance 100mm`
- **User Memory**
  - `UserRoom hasAvailableWidth 750mm`
  - `User owns ProductB`
- **Tool/API Retrieval**
  - `ProductA currentStatus Active`
  - `ProductA availableRegion Korea`

이 결과는 다음처럼 하나의 문맥으로 결합된다.

```
C(q) = Fuse( R_vect(q), R_graph(q), R_tool(q), M_user )
```

- `R_vect` : 벡터 검색 결과
- `R_graph` : 그래프 검색 결과
- `R_tool` : 외부 API·도구 실행 결과
- `M_user` : 사용자 및 세션 메모리

LLM은 이 통합 문맥을 바탕으로 최종 답변을 생성한다.

## 6. 기존 HybridRAG와의 차이

HybridRAG 역시 벡터 검색과 지식 그래프 검색을 결합한다. 기존 연구는 VectorRAG와 GraphRAG를 함께 쓰면 각각 단독 사용보다 검색·답변 성능이 높아진다고 제안했고, Microsoft GraphRAG도 비정형 텍스트에서 지식 그래프와 커뮤니티 계층을 만들어 검색·요약에 활용한다.

하지만 이 논문은 단순 검색 결합을 넘어선다.

| 일반 HybridRAG | 이 논문 |
| --- | --- |
| 벡터와 그래프를 함께 검색 | 벡터와 그래프를 서로 다른 **기억 계층**으로 운영 |
| 검색·답변 정확도 향상 중심 | 장기 기억·검증·계획·상태 관리 중심 |
| 주로 문서에서 그래프 생성 | 문서·API·대화·로그에서 온톨로지 생성 |
| 검색 후 답변 생성으로 종료 | 답변에서 신규 사실을 다시 추출 |
| 그래프를 검색 인덱스로 활용 | 그래프를 **지속적으로 갱신되는 세계모델**로 활용 |
| 엔티티·관계 검색 중심 | RDF/OWL·SHACL·Reasoner 검증 포함 |

> 정확한 핵심 문장: 이 논문은 벡터 검색과 그래프 검색을 단순히 병렬로 사용하는 데 그치지 않고, **텍스트 기억과 구조화된 세계 기억으로 역할을 분리**한 뒤 질의 시 다시 융합한다.

## 7. 두 병렬 파이프라인

**원문 기억 파이프라인**

> 문서·대화·API·로그 → 청크 분할 → 임베딩 → Vector Store → 의미 기반 검색

원문의 문맥과 세부 정보를 유지하고, 관련 텍스트를 빠르게 찾는 역할을 담당한다.

**구조 기억 파이프라인**

> 문서·대화·API·로그 → 엔티티 추출 → 관계 추출 → 정규화·정렬 → RDF Triple 생성 → SHACL/OWL 검증 → Trusted Graph → 관계·규칙 기반 조회

비정형 정보를 명시적 관계와 규칙으로 변환한다.

# Part 2. Feedback Loop — 기억을 갱신하는 작동 원리

## 8. Feedback Loop가 없으면 생기는 문제

Feedback Loop가 없다면 시스템은 사실상 다음 구조에서 끝난다.

```
문서 수집 → Vector Store·Ontology Graph 생성 → 질문 검색 → 답변 생성
```

이 구조는 초기 데이터에 대해서는 작동하지만, 시간이 지나도 다음 변화가 반영되지 않는다.

- 사용자가 대화에서 새로 알려준 정보
- API를 통해 확인된 최신 상태
- 업무 처리 결과
- 규정이나 제품 상태의 변경
- LLM 답변 과정에서 발견된 새로운 관계
- 기존 그래프의 오류 수정

결국 온톨로지가 점점 오래된 정적 데이터베이스가 된다. Feedback Loop는 이를 다음 구조로 바꾼다.

```
검색 → 답변 생성 → 새로운 사실 후보 추출 → 검증 → 그래프 업데이트 → 다음 질문에서 재사용
```

즉, 시스템이 단순히 지식을 조회하는 것이 아니라 **상태를 축적하고 갱신하는 장기기억 시스템**이 된다.

## 9. Feedback Loop 전체 흐름

논문의 파이프라인을 기준으로 정리하면 다음과 같다.

```
LLM Answer / Dialogue / Tool Result
                ↓
        Fact Extraction
                ↓
     Entity·Relation Detection
                ↓
   Normalization and Alignment
                ↓
      Candidate RDF Triples
                ↓
       SHACL/OWL Validation
         ↙              ↘
       실패              통과
        ↓                 ↓
   Logs에 기록       Trusted Graph 반영
                          ↓
                  다음 질의에 재사용
```

답변에서 추출된 사실은 다시 `Normalization and Alignment` 단계로 돌아가 RDF 트리플로 변환되고, **검증을 통과한 경우에만** 그래프에 기록된다.

## 10. 핵심: "답변을 그대로 저장하지 않는다"

Feedback Loop를 단순히 대화 내용을 DB에 저장하는 기능으로 이해하면 안 된다. 일반적인 채팅 메모리는 문장을 그대로 저장한다.

```
User: "Product A는 다음 달부터 단종이야."
Chat Memory: 해당 문장을 그대로 저장
```

논문의 구조에서는 이 문장을 **구조화된 후보 지식**으로 변환한다.

```
ProductA hasStatus Discontinued
ProductA validFrom 2026-08-01
Source     Conversation_1024
```

하지만 이것도 바로 신뢰 그래프에 들어가지 않는다.

```
사용자 발언 → 사실 후보 → 정규화 → 출처·형식·논리 검증 → 승인된 사실만 저장
```

<div style="margin:16px 0; padding:12px 16px; border-left:4px solid #ef4444; border-radius:8px; background:#fef2f2; color:#991b1b; line-height:1.6; text-align:center;">
<strong>Generated / Extracted Fact ≠ Trusted Knowledge</strong>
</div>

이 차이가 Feedback Loop의 핵심이다.

## 11. 실제 예시 — 상태를 가진 외부 메모리

사용자가 묻는다.

> "Product A를 우리 집에 설치할 수 있고 기존 Product B와 호환돼?"

시스템은 기존 이중 저장소에서 정보를 조회한다.

```
# 기존 그래프 정보
ProductA hasWidth 680mm
ProductA compatibleWith ProductB
ProductA requiresClearance 100mm

# 사용자 메모리
UserRoom hasAvailableWidth 750mm
User     owns ProductB
```

시스템은 계산 후 답한다.

> 설치 가능하며 Product B와 호환됩니다. 다만 최소 여유 공간 100mm가 필요합니다.

이후 사용자가 말한다.

> "실제로 설치했는데 오른쪽 공간이 80mm밖에 안 나와."

Feedback Loop는 이를 다음 후보로 변환할 수 있다.

```
Installation_204 hasProduct        ProductA
Installation_204 locatedAt         UserRoom
Installation_204 hasActualClearance 80mm
Installation_204 hasStatus         ConstraintViolation
```

이 사실이 검증되면 다음 질문에서는 단순 제품 스펙이 아니라 실제 설치 상태까지 반영할 수 있다.

```
기존 답변:      이론상 설치 가능
업데이트 후 답변: 설치는 완료됐지만 실제 여유 공간이 요구 조건보다 20mm 부족함
```

이것이 논문이 말하는 **상태를 가진 외부 메모리**다.

## 12. Feedback Loop의 세 가지 기능

### 12-1. 장기 기억 업데이트

새로운 대화와 사건을 세계 상태에 반영한다. 과거 문장을 추가하는 것이 아니라 **현재 상태를 갱신**한다.

```
ProjectA status InProgress
        ↓ (배포 완료 정보 유입)
ProjectA status         Deployed
ProjectA deploymentDate 2026-07-13
```

### 12-2. 생성 결과 검증

LLM 답변에서 사실을 추출한 뒤 기존 그래프와 비교한다.

```
기존 그래프: ProductA status Discontinued
LLM 답변:   ProductA is currently available
```

시스템은 두 사실의 충돌을 감지하고 답변을 수정하거나 `CONTRADICTED` 상태를 부여할 수 있다. 논문은 이를 생성과 판정을 분리한 두 루프로 본다.

```
Generation Loop:   LLM이 답변 후보 생성
Verification Loop: Ontology·Validator가 근거와 규칙을 검사
```

이 구조 때문에 시스템은 단순 RAG보다 **generation–verification–correction**에 가까워진다. ([arXiv](https://arxiv.org/abs/2604.20795))

### 12-3. 온톨로지 자체의 확장

Feedback Loop는 인스턴스만 추가하는 것이 아니라 새로운 관계나 클래스 후보도 만들 수 있다. 여러 대화에서 반복적으로 다음이 발견된다고 하자.

```
ProductA requiresVentilationSpace
ProductB requiresVentilationSpace
ProductC requiresVentilationSpace
```

시스템은 `VentilationSensitiveProduct`라는 개념을 후보로 제안할 수 있다. 다만 클래스나 스키마 변경은 그래프 품질에 큰 영향을 주므로 자동 반영보다 별도 승인 절차가 필요하다. 기존 LLM 기반 온톨로지 구축 연구도 자동 생성 결과에 대해 human-in-the-loop 평가를 권장한다. ([arXiv](https://arxiv.org/abs/2403.08345))

## 13. SHACL/OWL이 Feedback Loop에서 중요한 이유

Feedback Loop는 잘못 설계하면 오히려 시스템을 오염시킨다. LLM이 잘못된 답변을 만들고 그 답변을 다시 사실로 저장하면 순환 오류가 생긴다.

```
잘못된 LLM 답변 → 그래프 저장 → 다음 검색의 근거로 사용
→ 더 확신에 찬 잘못된 답변 → 다시 그래프 강화
```

이를 막기 위해 검증 단계가 들어간다.

**SHACL 검증** — 후보 트리플이 정해진 구조를 따르는지 검사한다. W3C 정의에 따르면 SHACL은 RDF 데이터 그래프를 별도의 shapes graph에 정의된 조건과 비교해 검증하는 표준이다. ([W3C](https://www.w3.org/TR/shacl/))

```
# 예: 제약
Product는 반드시 productName을 가져야 함
width는 숫자여야 함
Installation은 정확히 하나의 installedProduct를 가져야 함

# 다음 후보는 데이터 타입 위반으로 거절
ProductA width "wide"
```

**OWL Reasoning** — 기존 지식과 논리적으로 충돌하는지 확인한다.

```
ActiveProduct disjointWith DiscontinuedProduct
ProductA type ActiveProduct
ProductA type DiscontinuedProduct   ← 모순 감지
```

다만 SHACL·OWL 검증을 통과했다고 현실의 사실성이 보장되는 것은 아니다. **구조와 논리적 정합성**이 확인됐다는 뜻이다. 따라서 출처 신뢰도와 최신성 검증도 별도로 필요하다.

## 14. Candidate Graph와 Trusted Graph를 분리해야 한다

논문을 실제 시스템으로 구현할 때는 그래프 하나만 두기보다 다음처럼 구분하는 것이 안전하다.

| 영역 | 의미 |
| --- | --- |
| **Raw Log** | 원문 대화, API 응답, LLM 답변 |
| **Candidate Graph** | LLM이 추출한 미검증 트리플 |
| **Validated Graph** | 구조·논리 검증을 통과한 트리플 |
| **Approved/Trusted Graph** | 정책 또는 사람의 승인까지 통과한 지식 |
| **Deprecated Graph** | 오래됐거나 대체된 사실 |

```
Candidate: ProductA hasStatus Discontinued
Validated: 형식과 타입은 정상
Approved:  공식 제품 API로 상태 확인 완료
```

이 구분이 없으면 "형식적으로 올바른 사실"과 "실제로 신뢰할 수 있는 사실"이 섞인다. RIGOR 연구도 생성된 온톨로지를 한 번에 병합하지 않고, provenance가 포함된 delta fragment를 생성한 뒤 judge-LLM으로 정제하고 핵심 온톨로지에 병합하는 반복 구조를 사용한다. ([arXiv](https://arxiv.org/abs/2506.01232))

## 15. Feedback Loop는 두 종류로 나눠야 한다

논문 내용을 더 정확히 정리하려면 Feedback Loop를 두 가지로 구분하면 좋다.

**① Memory Update Loop** — 새로운 사실과 상태를 그래프에 축적한다. (목적: 장기 기억·최신성)

```
대화·API·행동 결과 → 사실 추출 → 검증 → 그래프 업데이트
```

**② Answer Correction Loop** — 생성된 답변을 그래프·규칙으로 검증하고 수정한다. (목적: 답변 신뢰성·정확성)

```
LLM 답변 후보 → 그래프 사실과 비교 → Supported / Contradicted 판정 → 답변 수정
```

논문에서는 두 흐름이 하나의 아키텍처에 섞여 있지만, 분리해 설명하는 편이 이해하기 쉽다.

# Part 3. 정리

## 16. 기존 RAG와의 차이 — 선형 vs 순환

일반 RAG는 보통 선형 구조다.

```
Retrieve → Generate
```

이 논문은 순환 구조다.

```
Retrieve → Generate → Verify → Extract → Update → (Retrieve again)
```

그래서 이 시스템은 단순한 검색 시스템이라기보다 **지속적으로 상태를 갱신하는 지식 운영 시스템**에 가깝다.

| 일반 RAG | 이 논문 |
| --- | --- |
| 문서를 검색하고 답변 | 답변 후 신규 사실 재추출 |
| 검색 시점의 문맥만 사용 | 장기 상태를 축적 |
| 원본 데이터 갱신이 수동 | 대화·API·행동으로 그래프 갱신 |
| 답변 생성으로 종료 | 검증·수정·저장까지 반복 |
| 정적 지식 기반 | 지속적으로 변하는 세계모델 |

## 최종 핵심

<div style="margin:16px 0; padding:16px; border-left:4px solid #6366f1; border-radius:8px; background:#eef2ff; color:#3730a3; line-height:1.7;">
<strong>이중 저장</strong>은 지식을 텍스트 기억과 구조 기억으로 나눈다.<br>
<strong>Feedback Loop</strong>는 새 정보와 생성 결과를 다시 구조 기억에 반영한다.<br>
<strong>검증 계층</strong>은 이 순환 과정에서 오류가 장기 기억으로 고착되는 것을 막는다.
</div>

따라서 이 논문의 아키텍처는 단순한 HybridRAG가 아니라 다음 구조에 가깝다.

> **Hybrid Retrieval + Structured Long-Term Memory + Verification + Continuous Graph Update**
