# 글쓰기 파이프라인 (Node Note)

한 편의 글을 기획부터 공개까지 올리는 실전 순서를 정리한다. 기본 명령 레퍼런스는
[GUIDE.md](../GUIDE.md)에 있고, 이 문서는 **실제로 글을 쓸 때의 절차와 판단 기준**
(특히 이미지·다이어그램·차트·디자인, 그리고 Blogger의 함정)을 담는다.

## 한눈에 보는 파이프라인

```
① 자료·주제 정하기
   → ② new 로 파일 생성
   → ③ 본문 이식·작성 (front matter + 마크다운)
   → ④ 디자인 입히기 (콜아웃·카드·색 시스템)
   → ⑤ 시각 자료 (이미지 / HTML 다이어그램 / CSS 차트)
   → ⑥ preview 로 미리보기
   → ⑦ publish --draft (초안 업로드)
   → ⑧ 라이브로 검증 (필요 시)
   → ⑨ publish --publish (공개)
   → ⑩ git 커밋·푸시
```

핵심 원칙 세 가지:

- **편집은 로컬 `.md`에서만, 반영은 CLI로만.** Blogger 편집기(작성 모드)는 열지 않는다.
- **시각 요소는 자체 완결형으로.** 스크립트·외부 자원 금지, CSS는 스코프 격리.
- **같은 파일을 다시 `publish` 하면 중복 없이 업데이트된다** (`blogger_post_id` 덕분).

---

## ① 자료·주제 정하기

- 주제(type)를 고른다: `ai`(AI 기술) · `paper`(논문 리뷰) · `etc`(잡담·회고).
- 세부 분류는 폴더가 아니라 글의 `labels`로 한다.
- 초안 텍스트는 어디서 써도 된다(직접 작성, 다른 도구 등). 이 파이프라인은 그걸
  블로그 형식으로 옮기는 과정이다.

## ② 파일 생성

```bash
npm run blog -- new <type> "<제목>"
# 예: npm run blog -- new paper "Attention Is All You Need 리뷰"
```

`posts/<type>/YYYY-MM-DD-slug.md`가 생성된다.

## ③ 본문 이식·작성

- 파일 위 `---` 사이 front matter에서 **직접 채우는 것**: `title`, `labels`.
- **자동 관리(건드리지 말 것)**: `blogger_post_id`, `published_at`, `updated_at`.
- 외부 자료를 옮길 때 **맨 위 `# 제목`(H1)은 제거**한다 — 제목은 front matter가
  담당하므로 본문 H1은 중복이 된다.
- 본문은 일반 마크다운(제목·목록·표·코드블록·인용). 발행 시 자동으로 HTML로 변환된다.

## ④ 디자인 입히기

밋밋한 "흰 바탕 검은 글씨"를 피하려면 핵심 지점에 시각 요소를 넣는다.
**모든 스타일은 인라인 CSS로** 넣는다(가장 안전, `<style>`·클래스보다 우선).

재사용하는 패턴(스니펫 전체는 [GUIDE.md](../GUIDE.md) 11장):

- **요약 카드** — 글 맨 위 "한눈에 보기"(그라데이션 배경 + 핵심 3줄).
- **콜아웃 박스** — 정보/팁/경고/핵심. 좌측 색 바 + 옅은 배경.
- **주체별 색 통일** — 비교글이면 대상마다 색을 정해(예: A=파랑, B=초록, C=골드)
  카드·배지·차트에서 일관되게 쓴다. 색 외에 라벨·아이콘도 함께(색만으로 구분 금지).
- **프로필/결론 카드** — 목록이 길게 이어지는 구간을 색 헤더 카드로 묶는다.
- **수미상관** — 도입부에 "왜 썼나"(동기), 결론부에 그에 대한 회수를 두면 글이 닫힌다.

> 주의: 텍스트(숫자·라벨)는 **중립 잉크**로, 색은 옆의 마크(점·막대·바)가 나른다.
> 옅은 색 글씨는 대비가 낮아 안 보인다.

## ⑤ 시각 자료 — 결정 트리

이미지·그림·차트는 방식이 갈린다. **로컬 파일 경로로는 안 되고**, Blogger 본문은
스크립트를 막는다는 점이 핵심.

```
사진·스크린샷 같은 실제 이미지인가?
├─ 예 → (A) Blogger 편집기에서 이미지 업로드 (Blogger가 호스팅) 또는 외부 URL
└─ 아니오(도형·구조도·차트)
     ├─ 이미 HTML로 있음 → (B) HTML 다이어그램 임베드
     └─ 데이터 그래프 → (C) 순수 CSS 차트로 재작성
```

### (A) 실제 이미지

- 이 CLI는 **로컬 이미지를 자동 업로드하지 않는다.** 본문에 로컬 경로를 써도 안 뜬다.
- 방법: 초안을 올린 뒤 Blogger 편집기에서 이미지를 넣거나(그러면 호스팅됨), 이미 호스팅된
  URL을 `<img src="...">`로 넣는다.
- 본문에는 삽입 위치 표시(점선 박스 등)를 남겨 두면 편하다.

### (B) HTML 다이어그램 임베드

외부에서 만든 HTML 그림을 넣을 때:

1. `<!DOCTYPE html>`·`<head>` 전체 문서면 **본문 조각(`<body>` 내부)만** 추출한다.
2. **CSS를 고유 래퍼로 스코프 격리한다.** 예: 최상위를 `<div class="dsra">`로 감싸고,
   모든 선택자를 `.dsra …`로 접두 처리. 그러지 않으면 `h1`, `.page`, `.title` 같은
   일반 선택자가 **블로그 테마와 충돌해 페이지 전체를 망가뜨린다.**
3. 폭이 고정(예: 1536px)이라 크면 → 래퍼에 `transform: scale(.58)` + **음수 마진**으로
   차지하는 공간을 보정한다(`zoom`은 텍스트 겹침 버그가 있어 쓰지 않는다). 좁은 화면은
   래퍼에 `overflow-x:auto`로 좌우 스크롤.
4. `<style>`은 통과되지만, 가능하면 이 블록 하나로 끝내고 본문 다른 곳엔 인라인을 쓴다.

### (C) 데이터 차트

- **Chart.js·D3 등 자바스크립트 차트는 Blogger에서 안 그려진다**(스크립트 차단, 외부 CDN
  차단). `<canvas>`+`<script>`를 그대로 넣지 말 것.
- 같은 그래프를 **순수 CSS(인라인 스타일)로 재작성**한다. 막대는 `div`의 `height`,
  색은 `background`, 격자선은 `repeating-linear-gradient`로.
- dataviz 원칙: ①색약 안전 팔레트(검증), ②막대마다 값 라벨, ③범례 항상, ④대비 낮은 색은
  값 라벨/표로 보완, ⑤y축을 자르면 캡션에 명시.

## ⑥ 미리보기

```bash
npm run blog -- preview "<파일>" --no-open
# 만든 HTML을 Windows에서 열려면 프로젝트 안으로 복사해 두면 편하다:
#   .blog-preview/ 폴더 (gitignore됨)
```

로컬 미리보기는 정확하다. 단, `preview`가 입히는 스타일 일부는 미리보기 전용이라 실제
블로그와 다를 수 있으니, 최종 확인은 라이브에서 한다.

## ⑦ 초안 업로드

```bash
npm run blog -- publish "<파일>" --draft
```

- 처음이면 새 초안 생성 + `blogger_post_id` 자동 저장.
- 이미 ID가 있으면 **기존 초안을 수정**(중복 안 생김). 몇 번을 다시 올려도 안전.

## ⑧ 라이브로 검증 (필요 시)

- 복잡한 HTML/CSS는 **실제 페이지에서만 정확히** 보인다.
- Blogger **편집기 안의 미리보기는 신뢰하지 않는다** — 화살표를 `&#8594;`로 바꿔 보여주는
  등 착시가 있다(실제 저장본은 정상).
- 가장 확실한 검증: CLI로 공개 발행 후 실제 URL에서 확인.

## ⑨ 공개 발행

```bash
npm run blog -- publish "<파일>" --publish
```

- CLI로 발행하면 **편집기를 거치지 않으므로** 본문이 깨지지 않고, 로컬 `.md`의
  `status`·`published_at`도 자동으로 맞춰진다.
- Blogger 웹에서 "게시" 버튼으로 올려도 되지만, 그 경우 로컬 상태가 어긋난다.

## ⑩ 버전 관리

```bash
git add posts docs GUIDE.md assets
git commit -m "post: <제목>"
git push
```

`.env`·`credentials/`·`.blog-preview/`는 `.gitignore`로 제외된다.

## 함정 & 철칙 (이번에 배운 것)

| 함정 | 대응 |
| --- | --- |
| 로컬 이미지가 본문에 안 뜬다 | 자동 업로드 없음 → 편집기 업로드 또는 호스팅 URL |
| JS 차트(Chart.js)가 안 그려진다 | 순수 CSS 차트로 재작성 |
| 다이어그램 CSS가 블로그 전체를 깨뜨린다 | 모든 선택자를 고유 래퍼로 스코프 격리 |
| `zoom`으로 줄였더니 텍스트가 겹친다 | `transform: scale` + 음수 마진으로 공간 보정 |
| 편집기 미리보기에 `&#8594;`가 보인다 | 편집기 착시일 뿐, 실제 저장본은 `→` 정상 |
| **Blogger 작성(Compose) 편집기로 열어 저장** | **절대 금지** — `<style>`·복잡 HTML·특수문자를 깨뜨림. 편집은 로컬 `.md`→CLI로만 |
| 같은 글을 또 올리면 중복될까 | `blogger_post_id`가 있으면 업데이트됨(중복 없음) |

## 치트시트

```bash
npm run blog -- new <ai|paper|etc> "제목"        # 생성
npm run blog -- preview "<파일>" --no-open        # 미리보기
npm run blog -- publish "<파일>" --draft          # 초안
npm run blog -- publish "<파일>" --publish        # 공개
npm run blog -- update  "<파일>"                  # 기존 글 수정 반영
npm run blog -- status  "<파일>"                  # 상태 확인
```
